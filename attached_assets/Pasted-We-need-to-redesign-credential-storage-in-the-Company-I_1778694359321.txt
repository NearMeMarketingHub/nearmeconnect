We need to redesign credential storage in the Company Information Hub so it is actually protected and not just visually masked.

Current issue:
Credentials/login information are currently being handled in a way that is not truly secure. Masking them in the UI is not enough if the underlying stored data or standard API responses still expose plaintext values.

Goal:
Implement real protected credential management for the admin-side Company Information Hub.

Important clarification:
These credentials need to remain usable by Agency Admins, so simple one-way hashing is not the right solution for stored passwords that admins may need to retrieve later. This should be implemented as protected/reversible secure storage appropriate for admin credential management.

Requirements:
- credential values must not be stored or returned in plain text through normal onboarding/company info responses
- credentials should be stored in a protected way
- credential access must be Agency Admin only
- credentials should be masked by default in the UI
- if reveal/copy is supported, it must happen through a deliberate admin action
- credentials should be handled separately from normal company notes/profile/reference info
- preserve the ability for admins to add/update login/access details over time

Implementation guidance:
- use a proper protected storage approach for secrets that need to be retrievable by admins
- do not leave passwords embedded in the general onboarding payload
- separate credential endpoints/data handling from general onboarding/company info responses
- if existing plaintext credential data already exists, add a safe migration path
- make sure previously collected credentials are not silently lost
- keep this centered in the admin-side Company Information Hub

Important constraints:
- do not expose decrypted credential values in broad company info responses
- do not allow client-side users to access credential data
- do not rely on UI masking alone
- keep the implementation organized and auditable
- preserve normal company info editing for non-sensitive fields

Please provide:
1. the secure storage approach used
2. how credential data is now separated from general onboarding/company info
3. how admin-only access is enforced
4. how existing plaintext credential data was handled or migrated
5. whether reveal/copy behavior was added and how it is protected
6. a brief QA checklist covering:
   - add credential
   - edit credential
   - masked default display
   - admin reveal/copy if supported
   - client-side access blocked
   - general onboarding/company info response does not leak credential values
